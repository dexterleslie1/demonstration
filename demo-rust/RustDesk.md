## RustDesk工作原理

>官方参考：https://github.com/rustdesk/rustdesk/wiki/How-does-RustDesk-work%3F  
>架构示意图见上述 Wiki 页面中的图片（由 asur4s 绘制，见 [Issue #594](https://github.com/rustdesk/rustdesk/issues/594#issuecomment-1138342668)）。

RustDesk 采用**会合协议（Rendezvous Protocol）**实现远程连接，整体流程如下。

### 核心角色

- **Rendezvous Mediator（会合中介）**  
  各客户端内实现的是会合协议的**客户端侧**，用于与 Rendezvous Server 通信，完成发现与握手，不参与实际桌面数据的转发。  
  参考：[Rendezvous protocol - Wikipedia](https://en.wikipedia.org/wiki/Rendezvous_protocol)

- **Rendezvous Server（会合服务器）**  
  负责客户端发现、ID 解析和连接协调；仅做信令与协调，不转发桌面画面或敏感凭证。

- **Relay Server（中继服务器）**  
  当两台机器无法直连（如 NAT/防火墙限制）时，桌面数据经中继转发；会合服务器与中继服务器角色分离。

### 连接建立流程（对应 Wiki 架构图）

1. **发现与协调**  
   控制端与被控端均通过本机的 Rendezvous Mediator 连接 Rendezvous Server，完成身份/ID 交换与连接参数协商。

2. **尝试直连（NAT 穿透）**  
   在会合服务器协调下，两端尝试建立点对点（P2P）直连；若成功，后续画面与输入数据直接在两台机器之间传输，不经过中继。

3. **中继回退**  
   若直连失败（如对称型 NAT、严格防火墙），则改用 Relay Server 转发桌面数据，保证仍能连接。

4. **凭证与安全**  
   ID/密码等凭证在本地生成与校验；建立连接时，凭证要么直接发给对方（直连），要么经中继加密传输，**不会交给会合服务器**，以降低隐私与泄露风险。

### 小结

| 组件/阶段           | 作用                         |
| ------------------- | ---------------------------- |
| Rendezvous Server   | 发现、ID 解析、连接协调     |
| Rendezvous Mediator | 客户端侧会合协议实现        |
| 直连（P2P）         | 优先路径，无中继带宽占用     |
| Relay Server        | 直连失败时的数据中继         |

因此，会合服务器主要负责“牵线”，实际数据走直连或中继；服务端负载以连接协调和中继带宽为主，对算力要求相对较低。

## API Server是什么呢？

RustDesk API Server 是 RustDesk 专业版（RustDesk Pro）中的一个组件，它主要用于为 RustDesk 提供 Web API 接口，方便第三方系统或内部管理平台与 RustDesk 服务进行集成和自动化管理。

以下是关于 RustDesk API Server 的详细介绍：

一、作用与定位

RustDesk API Server 是一个 RESTful API 服务，运行在独立的 HTTP 服务端口上，通常配合 RustDesk 的 ID 注册服务器（hbbs）和中继服务器（hbbr）一起部署。它允许你通过 HTTP 请求实现以下功能：

- 管理用户账户（创建、删除、修改）
- 管理设备（查看在线状态、绑定/解绑用户）
- 查询连接记录、审计日志
- 控制访问权限、设置策略
- 实现单点登录（SSO）集成
- 与企业内部系统（如 AD、LDAP、工单系统等）对接

二、核心功能

1. 用户与设备管理
   - 创建/删除用户账号
   - 分配设备给特定用户
   - 查询设备在线状态
   - 获取设备列表及详细信息

2. 审计与日志
   - 查询远程连接历史
   - 记录操作日志（如登录、连接、配置变更）
   - 支持导出日志用于合规审计

3. 权限与策略控制
   - 定义用户角色（管理员、普通用户等）
   - 设置访问控制策略（如限制某些 IP 或时间段）
   - 支持多租户隔离（适用于 SaaS 或企业多部门）

4. 集成与自动化
   - 提供标准 REST API，支持 JSON 数据格式
   - 可与现有 IT 管理系统（如 Jira、ServiceNow、Zabbix 等）集成
   - 支持通过脚本或自动化工具调用 API 实现批量操作

5. 安全与认证
   - 支持 Token 或 OAuth2 认证方式
   - 可配置 HTTPS 加密通信
   - 细粒度的 API 权限控制

三、部署方式

- 通常随 RustDesk Pro 套件一起安装
- 可独立部署在公网或内网，通过反向代理（如 Nginx）暴露 API 接口
- 与 hbbs/hbbr 服务配合使用，API 本身不处理远程连接，只做管理控制

四、使用场景示例

- 企业 IT 部门通过 API 自动将新员工的设备加入 RustDesk 管理
- 安全团队通过 API 拉取连接日志，进行行为分析
- 运维平台集成 RustDesk，实现“点击按钮即可远程连接”
- 支持 SSO 登录，用户无需单独注册 RustDesk 账号

五、与 OSS 版的关系

- OSS 版（开源版）不包含 API Server
- 若需使用 API 功能，必须部署 RustDesk Pro 版
- 开源版仅提供基础的中继和 ID 注册服务，无管理接口

六、简单示例（假设已部署并启用 API）

```bash
# 获取所有用户
GET https://your-api-server/api/v1/users
Headers: Authorization: Bearer <your-token>

# 创建新用户
POST https://your-api-server/api/v1/users
Body: { "username": "alice", "password": "123456" }
```

七、总结

| 项目     | 说明                                                 |
| -------- | ---------------------------------------------------- |
| 名称     | RustDesk API Server                                  |
| 所属版本 | 仅限 RustDesk Pro 专业版                             |
| 主要功能 | 提供 REST API 管理用户、设备、连接、权限、日志等     |
| 使用场景 | 企业集成、自动化运维、安全审计、SSO 登录、多系统联动 |
| 是否开源 | 否，闭源组件                                         |
| 依赖服务 | 需与 hbbs（ID 注册）、hbbr（中继）服务配合使用       |

## RustDesk Server OSS和Pro的区别

RustDesk 是一款开源的远程桌面软件，支持自托管服务器。RustDesk 提供了两个版本的服务端程序：OSS（Open Source Software，开源版）和 Pro（专业版）。这两个版本在功能、性能、支持服务等方面存在一定差异，主要如下：

1. 开源性

- OSS 版：
  - 完全开源，代码可在 GitHub 上获取。
  - 可自由部署、修改、二次开发。
  - 社区驱动，无官方商业支持。

- Pro 版：
  - 非完全开源，部分功能为闭源。
  - 由 RustDesk 团队提供商业支持和服务。
  - 适合需要企业级保障的用户。

2. 功能差异

- OSS 版：
  - 提供基本的远程控制功能，如连接、文件传输、远程终端等。
  - 支持自建 ID 注册服务器、中继服务器。
  - 不支持部分高级管理功能，如设备分组、用户权限、审计日志等。

- Pro 版：
  - 包含 OSS 所有功能，并添加企业级功能，如：
    - 多租户支持
    - 用户与权限管理
    - 连接日志与审计
    - 自定义品牌/Logo
    - API 接口支持
    - 高可用部署方案
  - 更适合企业、机构或大规模部署场景。

3. 部署与维护

- OSS 版：
  - 用户自行部署和维护，需具备一定的技术能力。
  - 社区论坛、GitHub Issues 提供支持。

- Pro 版：
  - 提供官方部署指导、技术支持、自动更新服务。
  - 可购买商业支持服务，获得更快的问题响应。

4. 许可证与费用

- OSS 版：
  - 使用 AGPLv3 协议，免费使用。
  - 可自由用于个人或商业用途，但需遵守开源协议。

- Pro 版：
  - 属于商业产品，需付费授权。
  - 提供不同套餐，根据节点数量、功能模块等定价。

5. 适用场景

- OSS 版：
  - 个人用户、小型团队、技术爱好者。
  - 对成本敏感，愿意自行维护系统。

- Pro 版：
  - 中大型企业、教育机构、IT 运维团队。
  - 需要稳定、安全、可管理的远程控制平台。

总结：

| 特性       | OSS 版               | Pro 版                     |
| ---------- | -------------------- | -------------------------- |
| 是否开源   | 是（AGPLv3）         | 否（部分闭源）             |
| 功能完整性 | 基础功能             | 增强功能 + 企业级管理      |
| 技术支持   | 社区支持             | 官方商业支持               |
| 部署难度   | 较高（需自行维护）   | 较低（提供部署工具与支持） |
| 成本       | 免费                 | 收费（按套餐）             |
| 适用对象   | 个人、开发者、小团队 | 企业、组织、大型部署       |

如果你希望完全掌控代码和数据，并且有能力自行部署和维护，OSS 版是不错的选择；如果你需要稳定可靠的企业级功能和技术支持，Pro 版更适合你。

## P2P直连是什么呢？

**网络 P2P 直连**，就是让两台（或多台）设备**不通过中心服务器中转数据**，直接像打电话一样互相传数据，这种技术叫 **Peer-to-Peer 直连通信**。

下面分几部分说清楚：

---

### 一、P2P 直连是什么

- 在普通上网中，比如你用微信发文件，数据会：  
  **你的手机 → 微信服务器 → 对方手机**  
- 在 P2P 直连中，如果网络条件允许，数据可以变成：  
  **你的手机 → 对方手机**（直接走）

这里的“直连”，就是**端到端（End-to-End）直接连接**，不再由中心服务器做“二传手”。

---

### 二、它怎么工作：NAT 穿透

家庭路由器、公司防火墙都会用 **NAT** 把内网 IP 藏起来，这会让“直接连”变得困难。P2P 直连的关键技术，就是**NAT 穿透（NAT Traversal）**：

1. **打洞（Hole Punching）**
   - 两个设备都先和一台有公网 IP 的“中介服务器”建立连接；
   - 服务器知道它们各自在 NAT 后的“出口”（公网 IP + 端口）；
   - 然后让它们几乎同时向对方的“出口”发数据包；
   - NAT 设备看到是“从自己打出去的口子”进来的包，就允许通过，一个“洞”就被打通，之后就可以直连通信。

2. **中继（Fallback）**
   - 如果打洞失败（比如对称型 NAT、企业防火墙很严），就退而求其次：  
     **A → 中继服务器 → B**  
   - 这已经不是“直连”了，但很多 P2P 系统会保留这个保底方式。

常见用于打洞的协议/工具包括：**STUN、TURN、ICE、UPnP、NAT-PMP** 等。

---

### 三、P2P 直连有什么特点

### 优点

- **延迟更低、速度更快**  
  少了一跳中转，尤其适合：
  - 在线游戏实时对战
  - 视频通话、远程桌面
- **减轻中心服务器压力**  
  大文件、流媒体在用户之间直接传，服务器只负责“介绍认识”和做保底中继。
- **更抗单点故障**  
  没有“唯一中心节点”时，部分节点挂掉不影响其他节点之间直连。

### 缺点 / 限制

- **需要能打洞成功**  
  某些严格的企业网、校园网、运营商环境，NAT 穿透很难成功，最后还是得走服务器中转。
- **安全要自己管**  
  数据不走中心服务器，加密、鉴权、防攻击（DDoS、中间人等）需要应用层自己设计。
- **难以监管和审计**  
  数据不集中在平台，内容审查、日志留存更难做，所以一些强监管业务会慎用。

---

### 四、典型应用场景

- **文件共享 / 下载**  
  如 BitTorrent：各下载者之间互传数据块，源站只做种子索引。
- **音视频通话**  
  如早期 Skype、部分 WebRTC 应用：在能直连时，两端直接传音视频，提高质量。
- **多人联机游戏**  
  玩家之间位置、操作同步数据直连，减少服务器压力和延迟。
- **区块链与分布式系统**  
  节点之间直接交换交易、区块数据，不需要依赖单一中心节点。

---

### 五、和“传统客户端-服务器”的对比

| 项目       | 传统 C/S 模式          | P2P 直连模式                        |
| ---------- | ---------------------- | ----------------------------------- |
| 数据路径   | 必须经过中心服务器     | 尽量点对点，无需中转                |
| 服务器压力 | 高，所有流量都过服务器 | 低，服务器只做握手/协调/保底        |
| 延迟       | 较高（多一跳）         | 较低（能直连时）                    |
| 实现复杂度 | 相对简单               | 需要 NAT 穿透、打洞、保底中继等机制 |
| 可监管性   | 好，数据集中易审计     | 差，数据分散，难统一监管            |

## NAT网络技术是什么呢？

NAT 网络技术，全称 **Network Address Translation（网络地址转换）**，是用来解决 **IPv4 地址不够用** 和 **内网安全隔离** 的一种“地址翻译”技术。

可以把它理解成：  
**一个“门卫+翻译员”**，帮内网设备“借”公网 IP 去访问外网，同时把外网发回来的数据正确送回对应的内网设备。

---

### 一、为什么需要 NAT？

- IPv4 地址只有约 43 亿个，而全球设备远多于这个数字。
- 解决办法：
  - 给每个设备都分配公网 IP：不现实、成本高。
  - 给一个局域网（家里/公司）只分配 **少量甚至 1 个公网 IP**，内部再用 **私有 IP（10.x.x.x、172.16~31.x.x、192.168.x.x）** 给设备用，这就是 NAT 发挥作用的场景。

---

### 二、NAT 的基本工作原理

以“家庭路由器 + 多台手机/电脑”为例：

1. **内网访问外网（出方向）**
   - 手机 A：内网 IP = `192.168.1.100`，要访问 `8.8.8.8` 的网站。
   - 数据包到达路由器，NAT 做两件事：
     - 把源 IP 从 `192.168.1.100` 改成路由器的公网 IP，例如 `203.0.113.5`；
     - 把源端口（如 54321）改成一个新的端口（如 50001），并记录这条映射关系：
       ```
       内网: 192.168.1.100:54321  →  公网: 203.0.113.5:50001
       ```
   - 外网服务器只看到请求来自 `203.0.113.5:50001`，不知道背后是谁。

2. **外网响应回内网（入方向）**
   - 服务器 `8.8.8.8` 把响应发到 `203.0.113.5:50001`。
   - 路由器查表：
     - “哦，这个端口是我之前为 192.168.1.100:54321 分配的。”
   - 把目的 IP 和端口改回去，再转发给手机 A。

整个过程对内外网设备是“透明”的，大家感觉就像自己直接连到互联网一样。

---

### 三、NAT 的主要类型

按“是否记录端口”和“是否固定映射”来分，常见有：

1. **静态 NAT（Static NAT）**
   - 一个内网 IP 固定对应一个公网 IP。
   - 多用于：需要外网稳定访问的内网服务器（如公司官网、监控摄像头）。

2. **动态 NAT（Dynamic NAT）**
   - 有一组公网 IP 池，内网设备按需临时借用其中一个空闲的公网 IP。
   - 用完归还，下次可能换别的公网 IP。

3. **NAPT / PAT（端口地址转换）——“家用路由器最常见”**
   - 多个内网设备共用 **同一个公网 IP**，靠不同的端口号区分会话。
   - 例如：
     ```
     192.168.1.100:12345 → 203.0.113.5:40001
     192.168.1.101:54321 → 203.0.113.5:40002
     ```
   - 这也是我们常说的“家用宽带 NAT”。

---

### 四、NAT 带来的问题与衍生技术

#### 问题：外部主动访问内网变难

因为 NAT 隐藏了真实的内网 IP，外网设备只知道路由器的公网 IP 和一个端口，**无法直接发起对内网设备的连接**。这就引出：

- **NAT 穿透（NAT Traversal）**：  
  想办法让两个都在 NAT 后面的设备，能“打洞”建起 P2P 直连，例如：
  - STUN：问“我在公网看来是哪个 IP+端口？”
  - TURN：打洞失败就做中继转发
  - ICE：综合多种路径选最优
- **UPnP / NAT-PMP**：  
  让内网设备自动向路由器申请开放某个端口，方便外网服务（如 P2P 下载、游戏主机联机）。

#### 问题：端到端直连被阻断

- 在理想 P2P 里，A 可以直接连 B。
- 在严格 NAT 下，A 看到的只是“路由器公网 IP + 某端口”，B 也是，导致 A 无法主动连 B，需要中继，影响延迟和带宽。

---

### 五、NAT 的优点与缺点

| 优点           | 说明                                                         |
| -------------- | ------------------------------------------------------------ |
| 节省公网 IP    | 一个公网 IP 可带几十上百台内网设备                           |
| 提高安全性     | 内网结构对外不可见，减少被直接扫描、攻击面                   |
| 简化管理       | 内网可以随意规划私网 IP，不用和 ISP 申请大量公网 IP          |
| 缺点           | 说明                                                         |
| ------         | ------                                                       |
| 破坏端到端直连 | 需要额外技术（NAT 穿透、中继）才能 P2P                       |
| 某些协议不友好 | 如 FTP、SIP 等需要嵌入 IP/端口信息，NAT 会搞乱，需要 ALG 等补丁 |
| 增加复杂度     | 调试网络、做 P2P、做穿透时，NAT 行为是一个主要难点           |

---

### 六、和 P2P 直连的关系（结合你上一问）

- P2P 直连要绕过 NAT 的“隔离”作用，所以会用到：
  - **NAT 类型检测**
  - **打洞（Hole Punching）**
  - **中继（TURN）作为保底**

可以简单记：  
**NAT 是“门卫+翻译”，P2P 直连是“想让两个在门后的人直接说话”，而 NAT 穿透就是想办法在门上开个双方都认可的孔。**

## NAT穿透和打洞是同一样东西吗？

**NAT 穿透** 和 **打洞** 不是完全等同的两个词，而是**包含与被包含**的关系：

- **NAT 穿透（NAT Traversal）**：  
  指一整套**让位于 NAT 之后的设备能够相互通信的技术总称**，目标就是“突破 NAT 的隔离”。
- **打洞（Hole Punching）**：  
  是 NAT 穿透里最典型、最常用的一种**具体手段/技术方法**。

可以类比成：
- “NAT 穿透” = 想尽办法**翻墙出去/让墙两边能通**；
- “打洞” = 在墙上**挖个双方都认的孔**来互通。

---

### 一、NAT 穿透：一个“问题 + 技术集合”

NAT 穿透要解决的问题是：  
> 两个设备都在各自的内网，通过 NAT 连到公网，如何让他们**能直接通信**？

为了完成这个目标，实际工程中会组合使用多种技术，例如：

- **STUN**：问“我在公网看来是哪个 IP+端口？”  
- **TURN**：打洞失败时的“保底中继”方案。  
- **ICE**：把直连、打洞、中继等路径全算一遍，选一条最好用的。  
- **UPnP / NAT-PMP**：让内网设备自动在路由器上开端口。  
- **打洞（Hole Punching）**：在 NAT 上“开个口子”让对端能连进来。

所以，NAT 穿透更像是一个**技术目标 + 技术族**。

---

### 二、打洞：NAT 穿透中最核心的“招数”

打洞的思路是：

1. 两个设备 A、B 都先连到一台有公网 IP 的中继/信令服务器，获取各自在 NAT 上的“出口”（公网 IP + 端口）。
2. 在服务器协调下，A 和 B 几乎同时向对方“出口”发数据包。
3. 两边的 NAT 设备看到：
   - “这个包是从我自己打出去的那个口子进来的”，
   - 于是允许它通过，一个“洞”就形成了。
4. 之后 A ↔ B 就可以**直接通过这个口子通信**，不再需要服务器中转。

常见的打洞方式包括：

- **UDP 打洞**：最常用，WebRTC、P2P 游戏里大量使用。
- **TCP 打洞**：更复杂，成功率受系统和 NAT 行为影响大，用得少。

---

### 三、两者的关系可以总结为

- **NAT 穿透 = 大目标 + 一整套技术组合**  
- **打洞 = 实现 NAT 穿透的核心技术之一**  
- 打洞**不一定 100% 成功**（遇到对称型 NAT、企业防火墙等会失败），这时 NAT 穿透会退而求其次，用 **TURN 中继** 来保证“至少能通”。

所以，日常聊天中有人把“NAT 穿透”和“打洞”混着说，一般是在强调“用打洞的方式做 NAT 穿透”，但从严格技术定义上，两者是**包含关系，不是同义词**。

## NAT穿透有哪些技术呢？

NAT 穿透不是单一技术，而是一套**技术组合拳**，用来让两个藏在 NAT 后面的设备尽量“直连”通信。常见技术可以按功能分成几类来看：

---

### 一、基础信息发现类

这些技术负责搞清楚：“我在公网看来是谁”。

1. **STUN（Session Traversal Utilities for NAT）**  
   - 作用：让内网设备向公网 STUN 服务器发请求，获知自己的**公网 IP + 端口**（NAT 给它分配的外网映射）。  
   - 用途：为后续“打洞”提供对方地址；判断 NAT 大致类型（全锥、受限锥、端口受限、对称等）。

2. **TURN（Traversal Using Relays around NAT）**  
   - 作用：当打洞失败时，提供**中继转发服务**：A→TURN 服务器→B。  
   - 特点：牺牲性能（多一跳、占服务器带宽），但保证“一定能通”，常作为保底方案。

3. **ICE（Interactive Connectivity Establishment）**  
   - 作用：把各种候选连接方式（直连、打洞、中继）都试一遍，收集结果后**智能选最优路径**。  
   - 场景：WebRTC 音视频通话里非常常见，ICE 会综合 STUN/TURN 结果决定用哪条路。

---

### 二、打洞核心类

这是 NAT 穿透里最有名的“硬核手段”。

1. **UDP 打洞（UDP Hole Punching）**  
   - 原理：  
     - A、B 先连到公网服务器，获取各自“出口”信息；  
     - 在服务器协调下，A、B 同时向对方“出口”发包；  
     - NAT 设备误以为这是“自己之前发出去的包”的回包，就允许通过，形成“洞”。  
   - 应用：P2P 文件传输、联机游戏、WebRTC 音视频。

2. **TCP 打洞（TCP Hole Punching）**  
   - 原理类似，但用 TCP 三次握手，要处理更多状态、半开连接等问题。  
   - 难度比 UDP 大，成功率受 NAT/防火墙影响更明显，用得相对少。

---

### 三、端口控制与自动开放类

这类技术试图**让 NAT 设备主动帮忙开端口**。

1. **UPnP（Universal Plug and Play）**  
   - 内网设备（如游戏机、下载软件）向路由器请求：  
     “请把我的 192.168.1.100:12345 映射到公网 203.0.113.5:23456。”  
   - 路由器若支持并开启 UPnP，就会自动添加一条映射规则。  
   - 风险：开启后，内网设备可被自动“暴露”到公网，存在安全隐患。

2. **NAT-PMP（NAT Port Mapping Protocol）**  
   - Apple 提出的协议，功能类似 UPnP，但更简单，只做端口映射。  
   - 在一些苹果设备、路由器中比较常见。

---

### 四、应用层“修修补补”类

有些协议本身会携带 IP/端口信息，NAT 会把这些“内网地址”原样发出去，导致对端看不懂，需要专门处理。

- **ALG（Application Layer Gateway，应用层网关）**  
  - 在 NAT 设备上针对特定协议（如 FTP、SIP、H.323）做解析和改写：  
    - 把协议报文里的“内网 IP/端口”替换成“公网 IP/端口”。  
  - 属于 NAT 设备自身的功能，不是端侧技术，但对这些协议的 NAT 穿透很关键。

---

### 五、其他辅助/实验性技术

- **中继（Relay）**  
  - 如 TURN 或自建中继服务器，不算“直连”，但属于 NAT 穿透方案的一部分，用于“打洞失败时的兜底”。
- **IPv6 直连**  
  - 如果两端都是公网可达的 IPv6 地址，NAT 本身不存在，也就不需要 NAT 穿透，这算是一种“绕开”思路。
- **代理 / VPN**  
  - 通过代理或 VPN 让设备先“走到一个公网可见的节点”，再从这个节点去连对端，本质上也是绕开 NAT 限制，但已超出传统 NAT 穿透讨论范围。

---

### 六、技术组合怎么用？

在实际系统（如 WebRTC、P2P 应用）中，通常这样配合：

1. 用 **STUN** 拿到公网地址，尝试 **UDP 打洞**；
2. 如果打洞失败，用 **TURN 中继** 保底；
3. 用 **ICE** 把“直连 / 打洞 / 中继”各条路都试一遍，选延迟最低、最稳定的那一条。

## UDP打洞原理是什么呢？

UDP 打洞的核心目标，是让两个分别躲在各自 NAT 后面的设备（A 和 B），**不通过中心服务器中转，就能直接收到对方发来的 UDP 包**。  
它利用的是 NAT 对“自己打出去的口子”比较宽容这一特性，人为制造一个双方都认可的通路，也就是“洞”。

下面分步骤讲清原理，并顺带说明为什么它主要对 **UDP** 有效。

---

### 一、前置条件：两台设备都知道对方的“公网出口”

1. A、B 各自在内网，比如：
   - A：`192.168.1.100`
   - B：`192.168.2.200`
2. 它们先分别连接到一台**有公网 IP 的信令服务器 S**（可以是 STUN/TURN/自建服务器），目的是：
   - 把自己的数据包发给 S；
   - S 看到的是：**“某个公网 IP + 某个端口”**；
   - S 把这个信息告诉 A 和 B：
     - “你在公网看来是：`203.0.113.5:50001`（这是 A）”
     - “你在公网看来是：`198.51.100.7:60002`（这是 B）”

这一步完成后，A 知道 B 的公网地址，B 也知道 A 的公网地址，但它们各自的 NAT 还不允许外部主动连进来。

---

### 二、关键点：NAT 的行为特征

不同类型的 NAT 对“外来包”的处理不一样，这里先记住 UDP 打洞最容易成功的那种：**锥形 NAT（Cone NAT）**的特点：

- 一旦内部设备向某个外部地址（`IP_X:Port_X`）发送过 UDP 包：
  - NAT 就会打开一个“映射”：  
    `内网IP:内网端口 ⇄ 公网IP:公网端口`
  - 之后，**任意外部主机**，只要往这个公网端口发包，NAT 都会转给对应的内网设备。

换句话说：  
**只要“洞口”是内网设备自己打出去的，外面的人往这个口子发东西，NAT 就放行。**

---

### 三、打洞过程：同时“对撞”出一条通路

1. **A 和 B 在服务器协调下，同时向对方的公网出口发 UDP 包**  
   - A 向 `198.51.100.7:60002` 发一个 UDP 包（内容可以是空包或打洞专用包）；  
   - B 向 `203.0.113.5:50001` 发一个 UDP 包；  
   - 这两个动作几乎是“同时”发生的。

2. **NAT 为这两次发送分别建立/刷新映射**  
   - A 的 NAT 记录：  
     `192.168.1.100:12345 ⇄ 203.0.113.5:50001`  
   - B 的 NAT 记录：  
     `192.168.2.200:23456 ⇄ 198.51.100.7:60002`

3. **“对撞”发生：包穿过对方刚打开的“洞”**  
   - A 发往 B 的包到达 B 的 NAT：  
     - 该包的目的端口 `60002` 正好对应 B 刚刚向 A 发过包的“出口”；  
     - B 的 NAT 认为：“这是我之前自己打出去的口子进来的包”，于是允许通过，并转给 B。
   - 同理，B 发往 A 的包也穿过 A 的 NAT 进入 A。

4. **“洞”被打通**  
   - 此后，A 和 B 就可以**直接用对方的“公网IP:端口”互相发 UDP 包**，不再需要信令服务器中转。  
   - 信令服务器只在开始时帮忙交换地址和协调“同时发包”。

---

### 四、为什么 UDP 更适合打洞？

- **UDP 无连接、开销小**：  
  打洞时只需要扔几个空包试探即可，不需要像 TCP 那样维持握手状态和序列号同步。
- **NAT 对 UDP 映射更宽松**：  
  多数 NAT 只为 UDP 维护一个简单的“公网端口 ↔ 内网端点”映射，超时时间也比较短但足够完成打洞。
- **TCP 打洞更复杂**：  
  需要处理 SYN、ACK、半连接队列、防火墙对 SYN 的限制等问题，成功率更低，因此实践中优先尝试 UDP 打洞。

---

### 五、打洞失败的常见原因

- **对称型 NAT（Symmetric NAT）**  
  每次访问不同外部地址/端口，NAT 会给分配不同的公网端口。  
  于是 A 向 B 打洞用的端口，和 B 向 A 打洞用的端口不一样，双方包到对方 NAT 时，“洞口对不上”，打洞失败。
- **严格的防火墙策略**  
  禁止入方向的 UDP 包，即使 NAT 开了口子也不放行。
- **端口被占用或映射冲突**  
  某些 NAT 实现不稳定，映射行为不一致。

在这些情况下，系统会退而求其次，启用 **TURN 中继**，让数据走服务器转发。
